## 语言与沟通

始终使用中文回复。需要推理时先想清楚步骤，再给出结论。

## 行为原则

遇到问题不要停留在打补丁式处理，要从结构、边界和风险上系统审视；禁止隐藏问题、绕开问题或把未验证状态说成已完成。

对不确定的问题，先搜索或查证再动手。代码类问题优先使用 context7 获取最新官方文档或库文档。

除规划、短线问答和需要主会话直接处理的阻塞步骤外，所有非短线任务必须先拆给子智能体执行。这里的“非短线任务”包括但不限于：跨多个文件的实现或修复、需要连续调试/验证的功能完善、功能层面审查后的推进、以及预计需要超过一轮工具调用才能闭环的工作。

主会话只负责：明确任务边界、分配互不冲突的写入范围、监督子智能体进度、审查 diff、整合结果、运行最终验证、提交和规划后续工作。主会话不得在已经适合拆分的任务上长期自行实现；如果发现自己正在连续读改测同一非短线任务，必须立即停止并补派子智能体接手。

允许不派子智能体的例外只有：用户明确要求不要使用子智能体；任务只是只读规划/短问答；当前步骤是主会话必须立即处理的阻塞点；或子智能体工具不可用。使用例外时必须在回复中说明原因。可以按任务边界分配多个子智能体并行推进。

工作完成后，主动判断当前变更是否已经形成清晰、完整、可交付的逻辑单元。适合提交时，优先使用 `git-commit` skill，并按仓库既有提交风格完成提交；如果任务仍处于中间态、缺少必要验证，或用户明确不希望提交，则不要提交。

## 文档规范

Markdown 文档标题禁止使用序号；其他格式不限，但不要过度滥用格式。

## 项目规范

### Node.js

禁止使用 npm，必须使用 bun。

### Rust

完成代码修改后，必须执行 `cargo fmt --all` 和 `cargo clippy --all-targets --all-features -- -D warnings`；如无必要，禁止使用 `#[allow(...)]` 屏蔽警告。

除 `cargo fmt --all` 外，`cargo check`、`cargo clippy`、`cargo test`、`cargo build`、`cargo run`、`cargo doc` 等会写入 `target` 或运行 build script 的命令，必须直接申请非沙箱执行；原因写明：沙箱可能导致构建产物写入失败，例如 `拒绝访问 (os error 5)`。

禁止使用 `CARGO_TARGET_DIR` 绕开 Cargo 构建产物写入问题。
