# 中文标点规范化

## 概览

规范中文正文里的标点，同时保护代码、公式、配置、路径、命令和其他机器可读语法。优先使用有边界的检查和替换，不做整篇无差别替换。

## 核心规则

- 中文字符之间使用中文标点：`，`、`：`、`；`、`？`、`！`、`（）`。
- 代码块、行内代码、命令、URL、文件路径、JSON/YAML/TOML、公式、API 参数和配置语法中的英文标点保持不变。
- 引号必须人工判断，不要盲目转换含中文的字符串，因为它们经常出现在代码或结构化数据里。
- 批量替换后必须查看 diff；文档有构建或预览步骤时，替换后要验证文档仍能正常生成。

## 常见误用红线

- 不要整篇无差别替换英文标点。
- 不要修改代码块、行内代码、命令、URL、路径、配置、公式或结构化数据。
- 不要盲目转换英文引号；含中文字符串可能是代码、JSON-like 数据或展示语法。
- 不要只搜索一种标点就结束；逗号、冒号、分号、问号、感叹号、括号都要按范围检查。
- 批量替换后必须看 diff；无法验证文档构建或预览时要说明原因。

## 快速对照

| 原始形式 | 目标形式 | 说明 |
| --- | --- | --- |
| `中文,中文` | `中文，中文` | 正文逗号 |
| `中文:中文` | `中文：中文` | 正文冒号 |
| `中文;中文` | `中文；中文` | 正文分号 |
| `中文?` | `中文？` | 中文问句 |
| `中文!` | `中文！` | 中文感叹句 |
| `(中文)` | `（中文）` | 仅限正文，不处理代码或配置 |

## 工作流

### 判断文件类型

先确认内容类型。Markdown、Typst、普通文档和中文正文通常可以规范化；源码、配置、代码示例和生成文件需要缩小匹配范围或手动处理。

### 统计疑似问题

编辑前先搜索中文字符附近的英文标点。根据环境优先使用 `rg` 或 `Select-String`。

```powershell
Select-String -Path .\*.md -Pattern '[\u4e00-\u9fff],[\u4e00-\u9fff]'
Select-String -Path .\*.md -Pattern '[\u4e00-\u9fff]:[\u4e00-\u9fff]'
```

常用替换模式：

```text
([\u4e00-\u9fff]),([\u4e00-\u9fff]) -> $1，$2
([\u4e00-\u9fff]):([\u4e00-\u9fff]) -> $1：$2
([\u4e00-\u9fff]);([\u4e00-\u9fff]) -> $1；$2
([\u4e00-\u9fff])\? -> $1？
([\u4e00-\u9fff])! -> $1！
```

如果替换工具只处理非重叠匹配，`中,中,中` 这类连续情况需要重复执行逗号替换，直到不再命中。

### 手动处理引号

先搜索含中文的英文引号，再逐项判断：

```powershell
Select-String -Path .\*.md -Pattern '"[^"]*[\u4e00-\u9fff][^"]*"'
```

函数参数、数组元素、JSON-like 值、代码示例和展示语法的注释里的引号通常不改。

### 复核例外

以下场景保留英文标点：

- 行内代码或 fenced code block。
- Typst 调用，如 `#num-table(...)`、`columns: (1fr, 3fr)`。
- 数学公式和技术记号。
- 英文缩写、英文名称或英文句子内部标点，例如 `AI,`。
- 列表标签、键值分隔符或格式本身要求的英文标点。

## 交付检查

- diff 中没有误改代码、配置、公式或路径。
- 预期正文匹配已经清理完，保留项都有合理原因。
- 文档有编译、构建或预览步骤时，已经完成验证。
