# 中文去 AI 腔

## 概览

把中文专业文本改得更像真实业务或技术人员写出来的文档。重点是去掉机械化表达、空泛强调和口号式总结，同时保留事实、约束、语气和文档用途；不要把专业文档改成闲聊文本。

## 核心规则

- 事实不变。不要编造数据、案例、结论、日期、人物或项目状态。
- 能具体就具体；没有依据的空泛判断要删除，或标注 `待补充` / `待确认`。
- 保持专业语气。自然不等于口语化。
- 除非用户明确要求大改，否则保留原文意图、受众和文档类型。
- 专业文档默认使用正向表述，优先写“负责什么、形成什么、支撑什么、推进什么、纳入什么管理”。否定表述只用于硬性禁止、风险提示、法律边界或安全边界。

## 常见误用红线

- 不要为了“自然”改掉原文事实、范围、责任边界或承诺强度。
- 不要把正式材料改成闲聊口吻、宣传口号或文学化表达。
- 不要补写看似合理但原文没有依据的数据、案例、成效或原因。
- 不要只替换几个连接词就交付；必须同时处理结构、句式和空泛判断。
- 不要把方案、规划、制度写成“不是、不再、不直接、不代表、不建议、不作为、不宜、避免”堆出来的排除清单。
- 不要用否定句承担段落主叙述。涉及不推进、不纳入、不适合的内容，应改成准入条件、优先级规则、后续评估条件或风险提示。
- 用户要求保留格式或标注改动时，必须优先满足该交付形式。

## 诊断清单

| 信号 | 常见形式 | 处理方式 |
| --- | --- | --- |
| 程式化开头 | `随着...的发展`、`在...背景下` | 直接写问题、动作、结果或结论 |
| 机械排序 | `首先`、`其次`、`再次`、`最后` | 用段落顺序、项目符号或自然转接 |
| 空洞强调 | `值得注意的是`、`需要指出的是` | 删除引导词，直接陈述内容 |
| 泛化评价 | `具有重要意义`、`发挥关键作用` | 改成具体影响、指标、依赖或风险 |
| 口号式结尾 | `综上所述`、`积极拥抱` | 改成实际结论、边界或下一步 |
| 节奏过齐 | 多句结构连续重复 | 调整句长和句式 |
| 抽象堆叠 | 多个 `的` 字短语连用 | 拆句，明确主体和动作 |
| 否定式边界 | `不再按...`、`不是...而是...`、`不直接...`、`不代表...`、`不建议...`、`不宜...`、`避免...` | 改成正向动作、准入条件、优先级规则或评估条件 |
| 内部批注口吻 | `泛泛写`、`只把...包装成...`、`优化口号`、`题目过散` | 改成正式判断，如“保护范围过宽”“技术机制不足”“布局方向分散” |

## 工作流

### 确认文档类型

先判断文本是报告、计划、验收结论、方案、通知、产品文案还是技术文档。改写风格必须服务具体文档类型，不套用同一套“自然化”模板。

### 拆掉程式化结构

- 背景铺垫式开头改为直接进入事实、结果、阻塞点或决策。
- 非必要的升华总结直接删除；文档确实需要结论时，只保留结论本身。
- 一个段落只讲一个重点；背景、动作、结论混在一起时要拆开。
- 只有在提升可读性时才使用项目符号，不为了显得整齐而列表化。

### 改成正向表述

- 方案、规划、制度类文本优先写正向动作：`负责`、`提供`、`形成`、`支撑`、`推进`、`纳入`、`聚焦`、`完善`、`建立`。
- `不是...而是...` 改为直接陈述目标、范围或组织方式；不要把修改痕迹写进正式文档。
- `不做/不建议/不宜/避免` 改为准入条件、优先级规则或风险提示。只有真正属于硬性禁止、法律边界、安全边界时，才保留否定句。
- `不代表/不直接等同于/不作为` 改为状态定义或管理规则，例如“纳入历史资产管理”“作为经验参考”“进入后续评估池”。
- 排除性清单应改成“纳入条件”“推进条件”“重点方向”“评估条件”；确需说明风险时，使用单独的风险提示段落。

### 改写句子

- 过载长句拆成短句，但不要切碎到失去专业感。
- 已知主体时，用“主体-动作-结果”替代抽象被动表达。
- 没有证据支撑时，删除 `显著`、`有效`、`持续` 等修饰词。
- 调整句子节奏，但不要添加夸张、煽动或文学化表达。

### 替换弱表达

| 避免 | 替换策略 |
| --- | --- |
| `首先/其次/再次/最后` | 直接按段落推进，或使用真实序号/条款 |
| `综上所述/总而言之` | 直接写结论 |
| `值得注意的是/需要指出的是` | 删除引导词 |
| `具有重要意义/发挥关键作用` | 写具体效果、指标、依赖或风险 |
| `不难发现/毋庸置疑` | 用证据表述，或直接陈述 |
| `不再按...而是...` | 直接写当前组织方式 |
| `不是...而是...` | 直接写目标或结论 |
| `不建议/不宜/避免` | 改成准入条件、优先级规则或评估条件 |
| `不代表/不直接等同于/不作为` | 改成状态定义、资产归类或管理规则 |

可使用 `但`、`所以`、`另外`、`同时`、`而`、`则` 等基础连接词，但只在它们能明确逻辑关系时使用。

### 核对内容完整性

改写后对照原文检查：

- 没有新增事实、数字、案例、范围或承诺。
- 没有弱化、夸大或改变原有结论。
- 不确定内容已删除或标注 `待补充` / `待确认`。
- 最终文本仍符合目标读者和文档类型。

## 示例

### 改写前

> 随着人工智能技术的快速发展，在企业数字化转型的背景下，智能客服系统的应用日益广泛。首先，它能够显著提升客户服务效率；其次，它可以有效降低人力成本；最后，它对于提升客户满意度具有重要意义。综上所述，企业应当积极拥抱这一技术变革。

### 改写后

> 智能客服系统已经进入企业客户服务流程。部署后，常见咨询可以由系统先行处理，人工客服主要负责复杂投诉、异常订单和高价值客户沟通。实际成效需要结合接入渠道、知识库质量和转人工比例评估，不能只按上线数量判断。

### 改写说明

| 改动 | 原因 |
| --- | --- |
| 删除背景铺垫 | 开头直接进入业务对象 |
| 删除机械排序 | 段落按服务流程推进 |
| 删除无依据指标 | 原文没有提供数据来源 |
| 补充评估条件 | 不编造事实的前提下让结论更具体 |
| 删除口号结尾 | 用可判断的边界替代号召 |

## 交付检查

- 没有残留不必要的程式化连接词或口号式结尾。
- 空泛形容词已替换为证据，或已经删除。
- 句子节奏自然，但仍保持专业文档语气。
- 缺少依据的内容已标注或移除，没有被补写成事实。
- 方案、规划、制度类文本没有把否定式边界当成主叙述；必要的否定表达已收敛到硬性禁止、风险提示、法律边界或安全边界。
