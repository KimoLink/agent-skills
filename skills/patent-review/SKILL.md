# 专利评审

## 概览

评审专利技术交底书和申请前技术材料，重点判断材料是否能支撑后续撰写权利要求和说明书。该技能不替代专利代理人或律师的法律意见；涉及授权前景、侵权判断、无效风险等法律结论时，只能标注为待专业确认。

## 适用场景

- 技术交底书、发明披露表、专利申请初稿的评审。
- 发明点、技术方案、实施例、附图说明、有益效果的完整性检查。
- 判断材料是否存在“只讲效果、不讲实现”“发明点分散”“权利要求支撑不足”等问题。
- 输出材料缺口、修改建议、可直接改写的技术表述；不要把评审变成向用户追问额外背景。

## 评审原则

- 先判断技术事实是否足够，再评价表达质量。
- 不编造现有技术、实验数据、性能指标、应用场景或法律结论。
- 区分“材料缺失”“表达不清”“保护范围风险”“需要专业确认”。
- 不把商业价值、管理流程、软件界面效果直接包装成技术贡献；必须落到技术问题、技术手段和技术效果。
- 一键审查默认只基于当前材料和可检索公开资料，不要求用户补充额外背景。
- 对不确定内容使用 `材料未体现`、`依据不足`、`需专业确认`，不要替用户补成事实。

## 评审流程

### 确认材料类型

先识别材料属于技术交底书、发明点说明、说明书草稿、权利要求草稿还是混合材料。不同材料的评审重点不同：

- 技术交底书：看信息是否足够支撑代理人撰写。
- 发明点说明：看创新点是否能归纳为技术特征组合。
- 说明书草稿：看技术方案、实施例和效果是否对应。
- 权利要求草稿：看保护范围、层级和说明书支撑。

### 梳理核心链条

按以下链条评审，不要只按文字顺序点评：

`现有技术缺陷 -> 技术问题 -> 技术方案 -> 关键技术特征 -> 实施例 -> 有益效果 -> 可扩展方案`

如果链条断裂，明确指出断点。例如：技术问题存在，但缺少对应技术手段；有益效果存在，但没有数据、逻辑或实施例支撑。

### 检查发明点

重点看：

- 发明点是否能用一两句话概括。
- 发明点是否体现技术特征组合，而不是业务目标或使用效果。
- 多个发明点之间是同一方案的层级关系，还是应拆分为多个申请方向。
- 是否存在可作为从属权利要求的改进特征、参数范围、替代结构、流程分支。

### 检查技术方案

技术方案至少要回答：

- 由哪些模块、步骤、数据、结构或算法组成。
- 各部分之间如何连接、传递、判断、控制或转换。
- 输入是什么，输出是什么，中间状态如何形成。
- 与现有技术相比，关键差异在哪里。
- 软件类方案是否说明数据处理逻辑、模型/规则、系统架构或流程边界。

### 检查实施例和支撑

实施例不是效果描述。检查是否包含：

- 具体实施步骤、结构组成、参数范围或运行流程。
- 至少一个完整可理解的实施方式。
- 与核心发明点对应的替代实施方式。
- 附图编号、部件编号、流程节点或数据字段说明。
- 有益效果的来源，必要时标注缺少测试数据或对比依据。

### 检查风险

常见风险要明确标出：

| 风险 | 表现 | 建议 |
| --- | --- | --- |
| 发明点空泛 | 只写提升效率、降低成本、智能化 | 补充具体技术手段和差异特征 |
| 支撑不足 | 权利要求或发明点比实施例更宽 | 增加实施例、替代方案和参数范围 |
| 现有技术不清 | 没说明已有方案怎么做、缺陷在哪 | 补充已知方案和具体不足 |
| 效果无来源 | 写显著提升、准确率提高但无依据 | 补充数据、逻辑推导或改为待验证 |
| 非技术信息过多 | 大量描述客户、价格、管理流程或界面效果 | 改成技术场景、技术手段和功能边界 |
| 单一实施例过窄 | 只描述一个界面或一个项目配置 | 补充通用结构、变形例和适用范围 |

## 交付检查

- 是否明确区分事实缺失、表达问题和需专业确认事项。
- 是否指出最影响申请撰写的阻塞点。
- 是否给出可执行的材料补充方向，而不是泛泛要求“补充细节”。
- 是否避免替用户编造现有技术、实验数据、效果指标或法律判断。
- 是否给出至少一处可直接采用的改写示例，除非用户只要求结论。
